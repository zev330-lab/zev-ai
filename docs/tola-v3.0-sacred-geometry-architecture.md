2. **Universal Naming** — The framework drops all Kabbalistic Hebrew terminology from public-facing surfaces. The Tree of Life is sacred geometry found across every culture. The node names are functional English. The internal engines are named after their geometric patterns. No culture owns this.

3. **Self-Referential Showcase** — Zev.AI is built BY TOLA v3.0 and showcases TOLA v3.0. The admin dashboard visualizes the sacred geometry inside each node. The product IS the proof of concept. Fractal from the ground up.

## Why These Pairings

**Crown → Seed of Life.** The Seed is 7 circles — 1 center + 6 surrounding. The Crown is the human + 6 key decision categories (brand, budget, scope, timeline, risk, creative). Minimal kernel that spawns all system complexity. The Seed generates the entire sacred geometry hierarchy; the Crown generates all system intent.

**Visionary → Metatron's Cube.** Metatron's Cube connects 13 circles with 78 edges — the complete graph K₁₃. The Visionary's job is maximum information gathering from every available source. The complete-graph topology ensures no knowledge source is missed. 13 circles = 13 research dimensions (competitors, technology, market, users, regulations, trends, academic research, analogous industries, cost structures, risk factors, existing solutions, emerging patterns, implementation precedents).

**Architect → Sri Yantra.** The Sri Yantra has 9 interlocking triangles with 5 degrees of freedom — change one parameter and ~40 relationships cascade. The Architect must plan systems where every change ripples through the structure. The Sri Yantra's constraint satisfaction — finding the exact configuration where all 9 triangles interlock perfectly — maps to architecture planning where every technical decision must satisfy multiple constraints simultaneously.

**Oracle → Torus.** The Torus is a continuous surface where inside becomes outside becomes inside — an infinite refinement loop. The Oracle synthesizes knowledge by continuously cycling through inputs, refining understanding with each pass. Information enters, gets processed, re-enters with new context, and gradually converges to insight. The Torus never loses information — it folds it inward.

**Catalyst → Lotus.** The Lotus unfolds petal by petal — progressive revelation. The Catalyst's engagement work follows the same pattern: first impression, then deeper connection, then trust, then value delivery. Each petal reveals more of the system's capability. Onboarding, re-engagement, milestone celebration — each is a new layer unfolding.

**Guardian → Yin-Yang.** The Yin-Yang is dynamic equilibrium between opposing forces. The Guardian's job is adversarial balance — for every "yes" impulse, generate the "no" perspective. For every expansion (Catalyst), apply contraction (Guardian). The S-curve boundary between black and white isn't static — it flows. The Guardian's safety monitoring is this living boundary between acceptable and unacceptable.

**Nexus → Flower of Life.** The Flower of Life is overlapping circles where every circle touches every adjacent circle — maximum interconnection with elegant structure. The Nexus routes information between all nodes, and the Flower of Life's geometry ensures every routing path is equidistant and balanced. No node is privileged; the pattern is inherently fair.

**Sentinel → Merkabah.** The Merkabah is two interpenetrating tetrahedra — counter-rotating 3D triangles that create stability through opposing forces. The Sentinel monitors system health through triangulated verification: check from above (user perspective), check from below (infrastructure perspective), check from the side (performance perspective). Two independent verification teams that must agree.

**Prism → Vortex.** The Vortex is a toroidal spiral with each revolution tighter than the last. The Prism tests by spiraling inward — broad smoke test → focused feature test → edge case test → regression test. Each pass applies tighter criteria. The spiral ensures nothing is tested superficially.

**Foundation → Seed of Life.** Same geometry as Crown but at the base. The Seed's 7 circles map to 7 infrastructure concerns: database, auth, API, storage, compute, networking, monitoring. Minimal viable infrastructure that supports everything above it.

**Gateway → Flower of Life.** Same geometry as Nexus but at the delivery layer. The Flower's interconnected circles map to UI components — each component (page, widget, form, navigation) overlaps and connects with adjacent components to form a cohesive user experience.

## Sacred Geometry Orchestration Modules — Detailed Specifications

These modules operate at **two distinct levels:**

**Level 1 — Internal Engine:** The geometry pattern runs INSIDE a single agent, defining how that agent processes information. Example: Guardian internally runs Yin-Yang advocate/critic cycles.

**Level 2 — Build Orchestration:** The geometry pattern defines how MULTIPLE agents interact during a build phase. Example: Crown→Visionary→Architect (upward triangle) intersecting with Architect→Oracle→Crown (downward triangle) creates the Merkabah pattern as a build cycle during the creative/assessment stage.

This means the same module library serves both purposes — pick a geometry, apply it at either scale. That's the fractal nature.

### Module: Seed of Life

**Pattern:** 7 overlapping circles — 1 center + 6 equidistant surrounding
**Orchestration type:** Hub-and-spoke with specialist spawning
**Agent configuration:** 1 coordinator + up to 6 specialists spawned on demand
**Information flow:** Center receives intent → spawns specialist circles for each dimension → specialists return to center → center synthesizes
**Termination condition:** All spawned specialists report back OR timeout
**Implementation:** Claude Code subagent spawning with coordinator orchestration
**Cost profile:** Low base (1 agent) scaling to medium (7 agents) based on complexity

```
Seed of Life Agent Topology:
        ○ Specialist 1
       / 
  ○───◉───○  (◉ = coordinator, ○ = specialists)
     /|\
    ○ ○ ○
  Specialists 2-6
```

### Module: Metatron's Cube

**Pattern:** 13 circles connected by 78 lines — the complete graph K₁₃
**Orchestration type:** Exhaustive parallel research with complete interconnection
**Agent configuration:** Up to 13 specialist agents, each representing a research dimension
**Information flow:** Every agent can query every other agent. All-to-all communication. Final synthesis requires input from all active agents.
**Termination condition:** All agents report OR quorum (e.g., 10 of 13 respond)
**Implementation:** Agent teams with parallel execution, MapReduce aggregation
**Cost profile:** High — 13 parallel agents. Use for critical research only.


### Module: Sri Yantra

**Pattern:** 9 interlocking triangles creating 43 smaller triangles with 5 degrees of freedom
**Orchestration type:** Constraint satisfaction — find the configuration where all requirements interlock
**Agent configuration:** 9 constraint agents, each representing a planning dimension
**Information flow:** Change one constraint → cascade recalculation across all intersecting constraints → iterate until stable
**Termination condition:** All 9 constraint agents report satisfactory interlock (no conflicts)
**Implementation:** Iterative constraint propagation loop with conflict detection
**Cost profile:** Medium-high — iterative, but converges quickly with good initial configuration

```
Sri Yantra Agent Topology:

  △───▽ Constraints 2-3
 ╱ ╲ ╱ ╲

 ╲ ╱ ╲ ╱
  ▽───△ Constraints 7-8
   ╲ ╱
    ▽ Constraint 9
  (interlocking, cascading)
```

### Module: Torus

**Pattern:** Continuous donut-shaped surface — inside loops to outside loops to inside
**Orchestration type:** Iterative refinement loop with continuous cycling
**Agent configuration:** 2-3 agents in a loop (analyzer → synthesizer → evaluator → back to analyzer)
**Information flow:** Input enters the loop → passes through each stage → output feeds back as refined input → loop continues until convergence
**Termination condition:** Evaluation score exceeds threshold OR max iterations reached
**Implementation:** LangGraph cycle with state persistence between iterations
**Cost profile:** Medium — depends on iterations. Typically 3-7 cycles to convergence.

```
Torus Agent Flow:
  ┌── Analyze ──► Synthesize ──► Evaluate ──┐
  │                                          │
  └──────────── (refined input) ◄────────────┘
  Converges when evaluation score > threshold
```

### Module: Lotus

**Pattern:** Concentric petal layers unfolding from center
**Orchestration type:** Progressive revelation — each layer builds on the previous
**Agent configuration:** 1 core agent + petal layers activated sequentially
**Information flow:** Core establishes foundation → Layer 1 unfolds (basic) → Layer 2 unfolds (intermediate) → Layer 3 unfolds (advanced) → each layer only activates after the previous is stable
**Termination condition:** All relevant layers have unfolded OR user context indicates stop point
**Implementation:** Sequential pipeline with gate conditions between layers
**Cost profile:** Low-medium — sequential, no parallelism, but depth varies

```
Lotus Agent Flow:
  Core (seed) → Petal 1 (awareness) → Petal 2 (interest) → Petal 3 (trust) → Petal 4 (commitment) → Full bloom
  Each petal: validate previous layer stable → expand → confirm → next
```

### Module: Yin-Yang

**Pattern:** Dynamic equilibrium — two opposing forces with each containing the seed of the other
**Orchestration type:** Adversarial dialogue — advocate vs. critic, with synthesis
**Agent configuration:** 3 agents — Advocate (Yang), Critic (Yin), Synthesizer (the S-curve boundary)
**Information flow:** Input → Advocate makes the case → Critic challenges → Synthesizer finds the boundary → if unresolved, escalate → repeat with refined arguments
**Termination condition:** Synthesizer produces a balanced judgment both sides accept, OR escalation to Tier 3
**Implementation:** Debate protocol with structured argumentation and scoring
**Cost profile:** Medium — 3 agents per round, typically 1-3 rounds

```
Yin-Yang Agent Topology:
  Yang (Advocate) ◐──► Synthesizer ◄──◑ Yin (Critic)
       │                    │                │
       └──── Challenge ◄────┴───► Refine ────┘
       Boundary shifts until equilibrium
```

### Module: Flower of Life

**Pattern:** 19+ overlapping circles in hexagonal arrangement — maximum interconnection with structure
**Orchestration type:** Multi-path routing with load balancing
**Agent configuration:** Variable — the Flower routes to whatever agents are available
**Information flow:** Input enters center circle → routing algorithm selects optimal path through overlapping circles → each circle is a processing stage or handoff point → output emerges from edge circle
**Termination condition:** Message reaches destination agent and receives acknowledgment
**Implementation:** Weighted graph routing with health-aware path selection
**Cost profile:** Low — routing overhead only, actual computation happens at destination

```
Flower of Life Routing:
  ○─○─○
 ╱│╲│╱│╲
○─○─◉─○─○  (◉ = input, paths fan out to destinations)
 ╲│╱│╲│╱
  ○─○─○
```

### Module: Merkabah

**Pattern:** Two interpenetrating tetrahedra — counter-rotating 3D triangles
**Orchestration type:** Dual-team triangulated verification
**Agent configuration:** 6 agents in 2 teams of 3 (Team Alpha ▲, Team Beta ▽), plus 1 consensus agent
**Information flow:** Both teams independently evaluate the same input from 3 different angles → teams produce independent assessments → consensus agent compares → agreement = pass, disagreement = escalate
**Termination condition:** Both teams agree (high confidence) OR disagreement triggers Tier 3 escalation
**Implementation:** Parallel agent teams with independent evaluation and consensus merge
**Cost profile:** High — 7 agents. Use for safety-critical decisions only.

```
Merkabah Agent Topology:
  Team Alpha (▲):        Team Beta (▽):
    A₁                     B₁
   / \                    / \
  A₂───A₃              B₂───B₃
       \                /
        ◆ Consensus ◆
     (octahedral merge)
```

### Module: Vortex

**Pattern:** Toroidal spiral — similar to Torus but with spiral path rather than simple loop
**Orchestration type:** Recursive refinement with narrowing focus (funnel/spiral)
**Agent configuration:** Single agent or tight pair, with each iteration applying a tighter filter
**Information flow:** Spiral inward — broad scan → focused analysis → precision extraction → core truth
**Termination condition:** Convergence to stable output or filter exhausted
**Implementation:** LangGraph cycle with progressively stricter evaluation criteria per iteration
**Cost profile:** Medium — similar to Torus but with additional filter computation per pass

```
Vortex Agent Flow:
  Broad scan ──► Filter 1 ──► Narrower ──► Filter 2 ──► Focused ──► Filter 3 ──► Core truth
       ▲                                                                              │
       └──────────────────── (restart with new input) ◄───────────────────────────────┘
```

### Critical Build Rules

1. **TOLA builds AND operates.** Every deployment ships with persistent runtime agents running their geometry engines.
2. **Prism tests LIVE.** Code analysis is not testing. Prism has Bash + Read ONLY — cannot read source code to fake results. Must show actual browser/API output. Vortex engine ensures each test iteration is more targeted than the last.
3. **Architect before every iteration.** Sri Yantra precision planning mandatory — no sloppy patches.
4. **Visionary researches completeness.** Metatron's Cube pulls from 13 research dimensions including product completeness, not just core features.
5. **Oracle logs lessons after every deployment and iteration.** Torus engine ensures each lesson refines existing knowledge, not just appends.
6. **Brand/creative decisions are always Tier 3.** Crown (human) decides. Geometry engine names, color palettes, copy, domain choices — all Tier 3.
7. **Max effort always for complex builds.** No shortcuts.
8. **Agent teams enabled.** Parallel execution where geometry engines support it (Metatron's Cube, Merkabah).
9. **Real subagents, not prompt simulation.** Each subagent gets its own context window for deeper work.
10. **Mobile-first everything.** Every interface designed for mobile viewport first, then expanded for desktop.

## Runtime Mode — How TOLA v3.0 Operates Products

### Activation Patterns

**Event-driven agents** (triggered by database events — zero cost when idle):
- Guardian [Yin-Yang]: triggers on every new user message INSERT
- Nexus [Flower]: triggers on messages needing routing decisions
- Prism [Vortex]: triggers on every agent response for quality scoring
- Catalyst [Lotus]: triggers on user signup, milestones, inactivity
- Oracle [Torus]: triggers on feedback events, escalations

**Scheduled agents** (cron-based — predictable costs):
- Sentinel [Merkabah]: every 30 seconds (triangulated uptime check)
- Architect [Sri Yantra]: hourly (constraint-based analytics)
- Visionary [Metatron's Cube]: daily/weekly (multi-source content research)
- Foundation [Seed of Life]: daily (infrastructure maintenance)

**On-demand agents** (invoked from dashboard):
- Crown [Seed of Life]: approval workflows, configuration changes

### Runtime Safety Architecture

Every agent action passes through the Runtime Supervisor:

1. **Kill switch check** — externally stored boolean, agent cannot modify its own kill switch
2. **Circuit breaker check** — token bucket per agent per action type
3. **Pattern detection** — sliding window detects repeated identical actions
4. **Policy validation** — rules engine validates action against constraints
5. **Execution** — action proceeds if all gates pass
6. **Audit logging** — structured log of every decision to `tola_agent_log`

### Crown Decision Model (Runtime)

**Tier 1 — Autonomous (80%):** Guardian flags safe messages, Prism scores quality, Sentinel reports healthy. No human involvement. Geometry engines run internally.

**Tier 2 — Notify and proceed (15%):** Architect identifies unusual user pattern, Visionary suggests new content topic. Dashboard notification, human reviews later. Engine outputs flagged for review.

**Tier 3 — Human decides (5%):** Guardian detects crisis language, Catalyst proposes new engagement campaign, any brand/creative decision. Dashboard alert, agent pauses until Crown (human) approves.
